# 数据处理方案对比：中央层 vs 压缩步长

## 📊 核心对比

### 架构图对比

```
【中央层方案（你批评的做法）】

Daloopa ─┐
         ├─→ 中央数据规范化引擎 ─→ 标准格式 ─┬→ Comps Analysis
FactSet ─┤   (复杂的转换逻辑)    (失去原始格式)  ├→ DCF Model
AKShare ─┤   (需要处理所有排列组合)             ├→ Portfolio
LSEG ────┘   (维护成本高)                      └→ Pitch Deck

问题：
❌ 中间层腐蚀：格式转换后丢失原始信息
❌ 复杂度爆炸：O(N×M)，N个源×M个目标
❌ 单点故障：中央层有bug影响全系统
❌ 维护困难：新增源/新增目标都需改中央层


【压缩步长方案（你建议的做法）】

Daloopa ┐
        ├─→ Comps Analysis ──→ (RULE 1转换)
FactSet ┤                      └→ 使用
AKShare ┤   Datapack Builder ─→ (格式检测+最小转换)
LSEG ───┴─→ DCF Model ────────→ (应用源头说明)
           Portfolio
           Pitch Deck

优点：
✅ 信息完整：源头格式始终可见
✅ 复杂度线性：O(N+M)
✅ 无单点故障：每个使用处独立
✅ 维护简单：新增源只需提供文档
```

## 🔍 详细对比表

### 1. 数据来源处理

| 维度 | 中央层方案 | 压缩步长方案 |
|------|---------|----------|
| **源格式明确性** | ❌ 隐藏在中央层 | ✅ 在源文档中明确 |
| **新增数据源** | ⚠️ 需修改中央层 | ✅ 只需提供源文档 |
| **源格式变化** | ⚠️ 需修改中央层 | ✅ 只需更新源文档 |
| **可追踪性** | ❌ 困难 | ✅ 容易（源引用) |
| **调试时间** | ⚠️ 需追踪中央层 | ✅ 直接看源文档 |

**例子：新增 Bloomberg 数据源**
```
中央层：
  需要在中央层添加：Bloomberg格式处理
  需要修改的地方：6+个使用处的映射表
  测试工作量：高

压缩步长：
  只需提供：Bloomberg 数据格式说明文档
  修改代码：各使用处根据说明独立处理
  测试工作量：低
```

---

### 2. 格式转换过程

| 维度 | 中央层方案 | 压缩步长方案 |
|------|---------|----------|
| **转换位置** | 中央集中 | 分散在使用处 |
| **源格式保留** | ❌ 丢失 | ✅ 保留 |
| **转换规则复杂度** | 高（处理所有组合） | 低（每处独立） |
| **维护位置** | 一个大文件 | 多个小文件+源文档 |
| **改动影响范围** | 全系统 | 该使用处 |

**例子：日期格式处理**
```
中央层方案：
  从源头获取：2024-01-31, 20240131, 01/31/2024, ...
  中央层规范化为：2024-01-31 (ISO 8601)
  使用时：所有使用处都用这个格式
  问题：如果源格式新增一种 → 中央层要改 → 全系统重新测试

压缩步长方案：
  Daloopa说明：返回 2024-01-31 (ISO 8601)
  AKShare说明：返回 20240131 (YYYYMMDD)
  Comps使用处：
    if source == 'Daloopa': use as-is
    elif source == 'AKShare': convert to YYYY-MM-DD
  新增源：只在该使用处添加转换逻辑，其他处不受影响
```

### 3. 可追踪性与调试

| 场景 | 中央层 | 压缩步长 |
|------|-------|---------|
| **看到一个数值，想知道来源** | ⚠️ "需要追踪中央层的转换日志" | ✅ "直接看源引用和颜色标记" |
| **数据不对，要找bug** | ⚠️ "可能在中央层的某个转换" | ✅ "直接看使用处的转换规则和源文档" |
| **需要验证数据准确性** | ⚠️ "需要理解中央层逻辑" | ✅ "对比源文档和原始值" |
| **需要重新计算某个指标** | ⚠️ "需要取消中央层转换再算" | ✅ "直接回到原始值算" |
| **需要向用户解释数据** | ⚠️ "需要解释多个转换步骤" | ✅ "直接说源和使用处的规则" |

**调试例子：Revenue数值异常**
```
中央层：
  用户：为什么DCF中Revenue是$1B，Comps中是$0.001B？
  工程师：让我查查... 中央层获取的是什么格式...
         AKShare是百万，Daloopa是千...
         中央层在某处转换错了？
         （调试时间：30分钟+）

压缩步长：
  用户：为什么DCF中Revenue是$1B，Comps中是$0.001B？
  工程师：看Comps的源引用...啊，这个用的是AKShare
         AKShare说明：返回百万 (不是$格式)
         Comps转换规则：应该乘以1M
         我来检查... 没乘！修复。
         （调试时间：3分钟）
```

### 4. 代码复杂度

| 维度 | 中央层方案 | 压缩步长方案 |
|------|---------|----------|
| **中央层代码行数** | 500-1000 行 | 0 行（无中央层） |
| **转换规则管理** | 中央配置表 | 源文档+使用处 |
| **测试覆盖** | 需覆盖所有源×目标组合 | 每个使用处独立测试 |
| **新增源后需修改** | 6+ 个地方 | 1-2 个地方 |

**代码行数估算：**
```
中央层（500+行）：
  def convert_ticker(ticker, from_source, to_source): ...
  def convert_date(date_str, from_format, to_format): ...
  def map_field_name(field, from_source): ...
  def normalize_currency(value, currency): ...
  def handle_missing_values(value, source): ...
  ... plus mapping tables ...

压缩步长（每处10-30行）：
  # Comps使用处
  for ticker in tickers:
      if source == 'AKShare':
          ticker = convert_astock_to_symbol(ticker)
      
  # DCF使用处
  if source == 'AKShare':
      dates = [d.to_date() for d in dates]
  
  # 各处都很简单，因为源文档明确了
```

### 6. 实际维护成本对比

```
【场景：从 Daloopa 迁移 30% 流量到 AKShare】

中央层方案的工作：
1. 修改中央层的日期格式处理 (Daloopa: YYYY-MM-DD → AKShare: YYYYMMDD)
2. 修改中央层的货币单位处理 (可能不同)
3. 修改中央层的缺失值处理 (可能不同)
4. 修改 6+ 个使用处的源检测逻辑
5. 测试所有可能的源组合 (现在是 2×N 而不是 N)
6. 更新文档说明
工作量：5-10 天
风险：高（改动范围大）
回退难度：难（涉及多个地方）


压缩步长方案的工作：
1. 在各使用处添加：
   if source == 'AKShare':
       dates = convert_dates(dates, 'YYYYMMDD', 'YYYY-MM-DD')
2. 测试各使用处独立（不需要测试源的所有组合）
3. 更新源文档和使用处注释
工作量：2-3 天
风险：低（改动范围小）
回退难度：容易（逐个使用处回退）
```

## 🎯 项目中的实际实现

### 当前项目（Datapack Builder）如何做

```markdown
【Phase 1: 原始提取】✅ 压缩步长实践
Step 1.1-1.5:
  ├─ 直接从源提取（保留格式信息）
  ├─ 记录页面引用（Daloopa p.47, FactSet p.3）
  └─ 不做初期转换

【Phase 2: 最小规范化】✅ 压缩步长实践
Step 2.1: 同源内规范化
  └─ 只确保同一来源的一致性，不做跨源转换

Step 2.2: 基于上下文的转换
  ├─ 读取：表标题、列标头、行标签
  └─ 原则：当不确定时，查看原始源文档
         (而不是查询中央层的转换表)

Step 2.3-2.4: 创建调整附表
  ├─ 记录：原始数据 → 什么调整 → 最终数据
  ├─ 完整可追踪性
  └─ 源引用明确：[Company] [Document] ([Date])

【Phase 3: 最终格式化】✅ 压缩步长实践
Step 3.2: 应用显示格式
  ├─ RULE 1: 财务数据 → $格式（基于数据含义）
  ├─ RULE 2: 运营数据 → 数字格式（无$）
  ├─ 触发词是"源头线索"（Revenue, Employees等）
  └─ 这是最后一步，不改变数据本身

【可追踪性】✅ 压缩步长实践
色彩编码：
  ├─ 蓝色：原始输入数据（用户输入或源直接）
  ├─ 黑色：转换后数据（公式计算）
  └─ 绿色：跨表引用
```

## 💡 你的观点的精妙之处

```
你说的：
"避免有一个复杂的层把所有的数据源的数据归一化，
 另外，如果做了数据处理，常常丢失原始数据的描述了，
 最后用的时候反而不知道格式了"

这精确指出了两个问题：
1️⃣  "复杂的中间层" → 违反KISS原则
2️⃣  "丢失原始数据描述" → 违反可追踪性原则

这正是为什么项目采用"压缩步长"的原因：
✅ 避免中间层
✅ 保留原始描述（源引用+调整附表）
✅ 使用处自己处理（基于源文档）
✅ 完整的数据谱系（源→调整→使用）
```
